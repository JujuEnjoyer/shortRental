# Безопасность и работа с персональными данными

Документ описывает прикладные меры защиты, реализованные в ShortRental, и ограничения дипломного прототипа.

## Модель доступа

В приложении используется роль `LANDLORD`. Все маршруты личного кабинета и API арендодателя требуют авторизации. Дополнительно проверяется принадлежность сущностей текущему арендодателю:

- объект недвижимости;
- бронирование;
- гость, связанный с бронированием;
- документ;
- фотография документа гостя.

Проверки выполняются на сервере, поэтому скрытие элементов интерфейса не является единственным механизмом защиты.

## Гостевые ссылки

Гость может открыть форму без регистрации по временному токену. Ссылка имеет срок действия. При истечении срока сервер возвращает ошибку, а арендодатель может выпустить новую ссылку.

Для защиты от перебора публичных маршрутов используется `RateLimitFilter`.

## Файлы и фотографии документов

Файловое хранилище не отдает фотографии документов по публичному URL. Фото паспорта доступны только через маршрут арендодателя:

```text
/api/landlord/guests/photos/{name}
```

Перед выдачей файла проверяется, что текущий арендодатель связан с гостем через бронирование.

`FileStorageService` выполняет:

- проверку расширения файла;
- запрет `/` и `\` в имени;
- нормализацию пути;
- проверку выхода за корневую директорию;
- удаление старых файлов при повторной загрузке.

## Внешние URL

ICS-календари и HTTP endpoint-ы интеграций проходят через `SafeExternalUrlValidator`.

Проверяется:

- схема `http` или `https`;
- наличие host;
- запрет `localhost`;
- запрет private/local network;
- таймауты подключения и чтения.

Это снижает риск SSRF и зависания синхронизации на недоступных ресурсах.

## Аудит действий

В системе ведется журнал значимых действий:

- создание бронирования;
- создание гостя;
- выпуск гостевой ссылки;
- заполнение данных гостем;
- загрузка фото документа;
- просмотр паспортных данных;
- подтверждение или отклонение паспорта;
- создание, подтверждение, отклонение и удаление документа;
- изменение политики хранения;
- запуск очистки устаревших данных.

Журнал нужен для контроля доступа к чувствительным данным и разбора спорных ситуаций.

## Политика хранения

Арендодатель может настроить:

- срок хранения паспортных данных после выезда;
- срок хранения сформированных PDF;
- необходимость удаления фото документов;
- флаг автоматической очистки для промышленного режима.

В демо очистка запускается вручную из интерфейса. Это сделано намеренно, чтобы случайно не удалить тестовые данные во время показа.

## Что требуется для промышленного внедрения

Для реального внедрения необходимо дополнить прототип:

- HTTPS;
- reverse proxy и WAF;
- резервное копирование;
- мониторинг и алерты;
- регламенты обработки персональных данных;
- согласие пользователя на обработку персональных данных;
- шифрование чувствительных полей при необходимости;
- безопасное хранение API-ключей во внешнем secret storage;
- ограничение доступа к серверной файловой системе на уровне ОС.

## Формулировка для защиты

Корректно говорить так:

> В дипломном проекте реализован прикладной контур безопасности: авторизация, роли, проверка владельца данных, временные токены гостевых ссылок, rate limiting, безопасная обработка файлов, проверка внешних URL, аудит действий и политика хранения данных. Сетевые атаки уровня DDoS должны дополнительно закрываться инфраструктурой: reverse proxy, WAF, CDN или настройками хостинга.
